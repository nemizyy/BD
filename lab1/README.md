## Короткий виклад вимог
**Потреби зацікавлених сторін**: Адміністрації, бухгалтерії та матеріально відповідальним особам установи необхідна зручна централізована система для інвентаризації та відстеження майна (від меблів до комп'ютерної техніки). Система повинна дозволяти закріплювати майно за конкретними працівниками, фіксувати його фізичне місцезнаходження (аудиторія/кабінет), групувати за категоріями для звітності та відстежувати поточний стан (в експлуатації, в ремонті, списано).

**Дані для зберігання:** Структура установи (відділи/кафедри), облікові записи співробітників (з прив'язкою до відділів), перелік фізичних приміщень установи, довідник категорій майна, а також детальна інформація про кожну одиницю майна (унікальний інвентарний номер, назва, вартість, дата взяття на баланс, статус, відповідальна особа та поточна локація).

**Бізнес-правила:** Кожна одиниця майна повинна мати унікальний інвентарний номер (дублікати заборонені). Майно не може існувати в системі без прив'язки до категорії, приміщення та матеріально відповідального працівника. Співробітник повинен обов'язково належати до одного з відділів установи. Вартість майна не може бути від'ємною.

## Сутності, їх атрибути та зв'язки
### Список сутностей та атрибутів:
Department (Відділ): id (PK), name (унікальна назва підрозділу), phone_ext (внутрішній телефон).

- Employee (Працівник): id (PK), department_id (FK), full_name (ПІБ), position (посада).

- Room (Приміщення): id (PK), room_number (номер кабінету, наприклад "405а"), building (назва або номер корпусу), capacity (місткість/площа).

- Category (Категорія): id (PK), name (назва категорії, наприклад "IT-обладнання"), depreciation_rate (відсоток амортизації).

- Asset (Майно): id (PK), inventory_number (унікальний номер, UQ), name (назва/модель), purchase_date (дата придбання), cost (початкова вартість), status (поточний стан), category_id (FK), room_id (FK), employee_id (FK).

Пояснення зв'язків прозою:
Department — Employee (Один до багатьох): В одному відділі чи на кафедрі працює багато співробітників, але кожен співробітник офіційно закріплений лише за одним підрозділом.

Employee — Asset (Один до багатьох): Один працівник може бути матеріально відповідальним за десятки або сотні одиниць майна (наприклад, системний адміністратор відповідає за всі комп'ютери), але кожна конкретна одиниця майна закріплена виключно за однією особою.

Room — Asset (Один до багатьох): В одному приміщенні знаходиться багато різного майна (столи, стільці, проєктори), проте одна фізична одиниця майна не може знаходитись у двох різних кабінетах одночасно.

Category — Asset (Один до багатьох): До певної категорії (наприклад, "Офісні меблі") належить багато одиниць інвентарю, але кожна річ класифікується лише однією категорією.

# Припущення та обмеження
- Одноосібна відповідальність: Зроблено припущення, що відповідальність за об'єкт не може бути колективною. Поле employee_id в таблиці Asset вказує на одну конкретну людину, з якої буде спитано під час інвентаризації.

- Облік історії переміщень: У поточній базовій моделі ми зберігаємо лише поточне місцезнаходження та поточного власника (перезаписуючи поля room_id та employee_id). Історія передачі майна від однієї особи до іншої або переміщення між кабінетами в цій ітерації не зберігається (для цього знадобилася б окрема таблиця логів).

- Обмеження на рівні словників (ENUM): Атрибут status у таблиці Asset обмежений фіксованим списком значень (наприклад: InUse (використовується), InStorage (на складі), InRepair (в ремонті), WrittenOff (списано)).

- Склад як приміщення: Припускається, що склади або підсобні приміщення також занесені до таблиці Room. Якщо майно ще не видане в кабінет, воно просто "знаходиться" в кімнаті з типом "Склад".

- Захист вартості: Припускається системне обмеження в базі даних (CHECK constraint), згідно з яким поле cost (вартість) у таблиці Asset повинно бути більше або дорівнювати нулю (cost >= 0).
